# Mockups and implementation workflow

Use this workflow only after the user has reviewed the references, evidence, and directions.

## Build code previews first

Use a token-first workflow. Inspect the project package manifests, routes, build commands, existing design tokens or theme sources, reusable components, assets, conventions, and dirty state before implementation. Build five isolated code previews—one route, story, screen, or entry per approved direction—using the same brief content and target viewports. Reuse the inspected token and component sources; disclose any necessary exception instead of inventing replacement brand values. If no suitable project exists, create one isolated Vite React TypeScript workspace with a shared standalone token layer and shared component primitives for all five directions.

Create `mockup-manifest.json` with exactly one `artifact_kind: code-preview` entry per approved direction using the fields in `artifact-contracts.md`. A successful local code preview may keep `attempt_count: 0`: no provider authorization or attempt is required. Render every target viewport, record source-bound screenshots and passing `viewport_checks`, validate all entries, then transition to `mockups_generated`. Present the five isolated code previews with their token/component provenance and ask the user to select or `revise`.

## Optional generated assets

Image generation is optional. Use it only when the user explicitly requests it or an approved direction needs a visual asset that project assets and code cannot express. Explain the asset and obtain explicit approval before any provider call. A generated image alone does not satisfy the code-preview requirement; integrate the approved asset into its isolated preview and rerender the local comparison screenshots.

Write a prompt file inside the run, then immediately before every image provider call run `python3 scripts/run_state.py can-generate --run <run-dir> --direction <id>`, followed by `python3 scripts/run_state.py authorize-generation --run <run-dir> --direction <id>`. Do not call the provider without a successful reservation. Authorization changes only the manifest through one atomic replace; run.json is unchanged. After the reserved call, keep the attempt, record `failed` or `success`, and store the typed reference `provider:<lowercase-provider>:<safe-artifact-id>` in `supporting_provider_refs`; provider artifact existence is the host's responsibility.

If a user-provided or internal screenshot may contain personal, confidential, or proprietary information, warn the user before sending it to an external image-generation provider. Redact it or obtain informed confirmation; never put credentials, cookies, API keys, or pairing tokens in prompts or artifacts. Follow the host's image-generation response rule and end the turn immediately after emitting generated images; record outputs and continue manifest/state work on the next turn. Permit one technical retry per failed direction; ask before more attempts or images. Record approved expansion with `--generation-budget`, `--max-attempts-per-direction`, and `--approve-budget-expansion`. Validation remains cumulative, and attempt totals must match.

Authorization first resolves the OS account record from `pwd.getpwuid(os.getuid()).pw_dir`, not `$HOME`, and opens the account home directory inode as the primary global generation mutex. The passwd path must be an absolute real directory owned by that account; its parent must be root-owned and not group- or world-writable, and home path-to-fd identity must match. If this stable namespace cannot be established, authorization fails closed. It holds a nonblocking POSIX flock on that home descriptor before inspecting the requested run. This creates a global serialization tradeoff: cooperating generation authorizations for every run and runs-root owned by the account are serialized.

After the primary lock, authorization opens and locks the canonical runs-root directory fd as a secondary global runs-root mutex, then locks the resolved run-directory inode. The stable primary prevents runs-root rename-clone ABA, while the runs-root and run-directory locks remain defense in depth against local path replacement. Path-to-fd identity is checked before mutation and after publication; all ledger staging and replacement is relative to the held run-directory descriptor. Release only unlocks and closes the three descriptors. On normal exit or SIGKILL the kernel releases the flock on each descriptor. Diagnostic lock metadata is non-authoritative: legacy .generation.lock is never read, truncated, unlinked, or replaced, so recreating it cannot admit another writer and a hardlink cannot expose a victim to writes.

Manifest staging loops over short writes, rejects a zero/invalid result, and fsyncs only after all bytes are written. The staged JSON is parsed and matched before replace while its descriptor stays open. Publication verifies the named stage still has that inode, performs its one atomic descriptor-relative replace and directory fsync, then does post-replace destination inode, byte, and JSON verification. A mismatch fails authorization and atomically restores the byte-identical prior ledger with another fsync. A failed or interrupted reservation leaves either the old valid manifest or the fully verified new one, never corrupt success.

The linearization point is successful post-replace verification and directory fsync. At that point, publication is atomic and durable under cooperating Design Explorer processes and crash/SIGKILL recovery. It cannot guarantee pathname contents remain unchanged against arbitrary same-UID mutation at any time; a process that ignores the protocol can alter account-owned content during or after authorization. Invalid external tampering is detected on the next load or validation, including `status`; an endless final reread would not extend the completed operation's guarantee.

## Select or revise

- **Select:** Record the selected approved ID by transitioning to `implementation_selected`, then build its isolated preview.
- **Revise:** Only from `mockups_generated`, run `python3 scripts/run_state.py revise --run <run-dir> --reason "<user-request>"`. The command revalidates the current mockups before any archive or state mutation, then uses the durable revision transaction marker and shared mutation locks to archive them as `mockup-manifest.revision-<n>.json`, record the audit reason/time/count, clear selection and approvals, and return to `directions_pending_approval`. A later load recovers an interrupted transaction before normal state validation.
- Append only the user-requested number of bounded derived directions to `directions.json`, after their sources. Give each a unique, first-class `id` and `kind: derived`; never reuse an arbitrary primary direction ID. Record unique, non-empty `derived_from_ids` containing only prior direction IDs. Record a non-empty `combined_properties` object whose keys are limited to the six axes and whose values name IDs in `derived_from_ids`. Each mapped derived axis must equal its named source's axis after trim/case normalization, and every source ID must contribute at least one axis. Retain the complete direction contract, including linked official evidence, `baseline_exceptions`, all six axes, risks, and trade-offs. A new direction must still pass the three-axis pairwise diversity rule.
- Update `mood-directions.md`, disclose evidence and baseline exceptions, validate directions, present the derived direction, and obtain explicit approval again. Do not generate its image before the new ID is recorded in `directions_approved`.
- Create a fresh `mockup-manifest.json` containing only the newly approved IDs. Generation accounting resets through absence, creation, and archival of the manifest; it is never copied into `run.json`. `revise` resets generation and attempt budgets to defaults and clears prior expansion approval. Generate at most the bounded variations the user authorized; record fresh explicit approval before any expansion.

## Build an isolated preview

1. After selection, inspect the active repository's package manifests, routes, components, design tokens, scripts, and dirty state. Preserve unrelated user changes and use the project's existing stack and conventions.
2. Add an isolated route, screen, story, or component that cannot replace or silently affect the production screen. During exploration, never overwrite production files or production navigation.
3. If no suitable project exists, create a standalone Vite React TypeScript preview in an isolated directory. Keep its dependencies and entry point self-contained.
4. Reproduce the selected first-class direction with real UI structure and the brief's required content. For a derived direction, follow its recorded `combined_properties` rather than recombining informally.
5. At every target viewport, capture a screenshot and compare the rendered screen against each required brief content item. CSS-hidden content passes only through an operable, tested disclosure path. Controls labeled `Edit`, `Change`, or `Apply` must function or must not be presented as a control. Check visual hierarchy, overflow, responsiveness, keyboard/focus behavior, semantic structure, contrast, and accessible names in the rendered and interactive state, not DOM presence. Run relevant lint, typecheck, and tests.
6. Write `implementation.json` with `selected_direction_id`, `mode`, `preview_path`, non-empty `preview_files`, normalized absolute `preview_route`, and `verification`. A project preview also records `route_registry_path` and `route_consumer_path`; the structured registry entry has exact `component_path` and `shell_id` fields, and the isolated consumer imports both files without overwriting production. Project wiring requires specifier-level runtime bindings for both files: whole `import type`, inline `type` specifiers, and side-effect-only imports do not satisfy project wiring. The validator's lexer rejects imports/calls hidden in comments, template literals, or ordinary strings and binds standalone calls to actual imported bindings. A standalone preview uses the complete executable Vite React TypeScript shape in `artifact-contracts.md`, not placeholders.
7. Make `rendered_viewports` exactly match the run targets. Add `viewport_checks` keyed by every viewport. Each record contains passing status and exact required-item maps, a `screenshot_ref` to a PNG stored inside the run directory, and `source_digest` over canonical sorted preview file paths and bytes. Include recursive code, CSS, HTML, `new URL`, and direct or brace-wrapped JSX literals in `src`, `href`, `poster`, and `srcSet`; every local dependency must be contained and listed. From a mixed data URI and local value, extract each explicit `./` or `../` `srcSet` candidate, ignore remote/data entries, and do not treat arbitrary prose as an asset. Runtime-computed and template-literal imports are out of scope and require explicit inclusion plus runtime verification. Recompute screenshots and digest after any source change. PNG validation covers complete bounded chunk structure, every CRC, IDAT, terminal IEND, dimensions, truncation, and trailing bytes without pixel decoding. Aggregate checks cannot replace this evidence. Do not record `content: pass` while required content is inaccessible. Validate before `prototype_ready`.
8. Present the verified isolated preview for review. Use an in-process localhost HTTP registry-backed route probe for the registered route and a missing-route 404. It proves topology and route resolution, not browser pixel rendering. Integrate only after explicit preview approval, then use the state command's integration-approval flag.
